# 工程协作原则

本文从全局协作规范中挑选适合本插件的工程约束。它不是通用个人偏好全集，只记录会直接影响 `astrbot_plugin_status` 维护质量的规则。

## 工具与检索

- 能本地处理就先本地处理；确有必要时再联网。
- 先小范围定位，再精读、修改或扩大检索范围。
- 文本检索优先使用 `rg`。
- 大文件先检索命中位置，再按需分段读取。
- 命令必须显式设置工作目录，不依赖前一次 `cd` 的状态。
- 联网、安装依赖、长任务、破坏性动作或大规模重构前，先说明目的和影响。

## 任务收敛

- 大任务不要直接闷头实现，先澄清目标、边界、影响范围和验收标准。
- 单次改动尽量围绕一个目的；不要把 unrelated 重构塞进修复提交。
- 行为边界不清楚时，优先写计划或列出假设，再开始改代码。

## 代码组织

- 一个文件只承载单一职责或紧密相关职责。
- 文件开始变大、变杂、难导航时，应优先拆分。
- 优先沿用现有模块、命名和 helper，不为了局部问题发明新风格。
- 中文注释用于解释关键意图、不直观逻辑、约束条件和边界情况；不要写纯噪音注释。

## 限制与保护逻辑

- 不要为了“看起来更稳”而新增无依据的硬编码限制、截断、固定超时、重试上限、静默回退或隐藏兜底。
- 新增限制必须至少满足以下条件之一：
  - 产品、README 或现有文档明确要求
  - 平台、接口或运行环境存在客观限制
  - 沿用项目既有同类约束
  - 有测试或日志证据证明不加会稳定故障
- 新增超时、长度、条数、重试、截断、降级或回退逻辑时，要在代码或文档中说明触发条件、目的和影响。
- 不得让合法内容被静默截断，也不得把真实失败伪装成成功。

## 错误处理

- 错误处理目标是暴露真实问题并方便排查。
- 不要 broad `try/except` 后直接吞掉错误。
- 不要返回空结果冒充成功。
- 日志应尽量区分用户配置缺失、模型不可用、T2I 渲染失败、资源路径非法和系统指标读取失败。
- 降级路径必须尽量保持可观测，不能偷偷改变成功语义。

## 验证

- 代码任务完成时默认检查：测试是否补齐、回归是否执行、文档是否需要更新。
- 渲染、模板、图片资源或 LLM 分析改动不能只看代码，至少做语法检查和相关路径的最小手工验证计划。
- 本插件没有 Plugin Pages，前端验证原则不适用；HTML/CSS 模板改动关注 T2I 渲染结果即可。
